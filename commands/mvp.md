# MVP 기획 (대화형 + 전문가 패널)

**사용법**: `/mvp [주제]`

사용자와 대화로 아이디어를 구체화하고, 전문가 패널이 두 번 개입하여 MVP 기획서를 만듭니다.

## 인자

$ARGUMENTS

- 주제 문자열이 있으면 Phase 1의 첫 질문에 반영
- 주제가 없으면 "어떤 서비스를 만들고 싶어요?"로 대화 시작

## 전체 플로우

```
Phase 1: 대화형 기획 (질문-답변으로 윤곽)
    ↓
Phase 2: 1차 전문가 검토 (방향성 검증 + 아이디어 제안)
    ↓
Phase 3: 기획 심화 (전문가 피드백 반영, 스코프/플로우 확정)
    ↓
Phase 4: 기획 초안 작성 (기획서 + 플로우차트)
    ↓
Phase 5: 2차 전문가 보강 (완성도 검증 + 최종 보강)
    ↓
Phase 6: 최종 산출물 (기획서 확정, docs/mvp/에 저장)
    ↓
Phase 7: 다음 단계 선택 (화면설계 / 기술설계 / 구현 / 끝)
```

---

## Phase 1: 대화형 기획

**한 번에 하나씩** 질문하며 아이디어를 구체화한다.

### 신규 프로젝트인 경우

질문 가이드 (상황에 따라 추가/생략/순서 변경 가능):

1. "이 서비스를 한 문장으로 설명하면?" → 핵심 가치
2. "타겟 사용자가 누구예요?" → 페르소나
3. "사용자가 이 서비스에서 가장 먼저 하는 행동은?" → 유저 플로우 시작점
4. "비슷한 서비스나 레퍼런스가 있어요?" → 벤치마크 + 차별점
5. "MVP에서 반드시 있어야 하는 기능 3개만 뽑으면?" → 스코프 제한
6. "기술 스택 선호가 있어요? 없으면 추천해드림" → 기술 제약
7. "특별한 제약이 있어요? (예산, 기간, 플랫폼 등)" → 제약 조건
8. "성공을 어떻게 측정하고 싶어요?" → KPI

### 기존 프로젝트 확장인 경우

주제에서 기존 프로젝트 확장임이 감지되면 (예: "기존 서비스에 ~ 추가"):

1. 현재 디렉토리의 코드베이스 구조 스캔 (있는 경우)
2. "현재 서비스의 핵심 기능은 뭐예요?" → 기존 맥락
3. "추가하려는 기능이 기존 서비스와 어떻게 연결되나요?" → 통합 지점
4. 이후 신규 프로젝트와 동일한 질문 흐름

### 규칙

- 질문은 고정 목록이 아님. 상황에 따라 유연하게 조정
- 답변이 모호하면 후속 질문으로 파고듦
- 답변 중 이미 다른 질문의 답이 나왔으면 스킵
- 충분히 이해됐다고 판단하면 Phase 2로 넘어감

## Phase 2: 1차 전문가 검토 (방향성 검증)

대화로 윤곽이 잡힌 시점에 전문가 패널을 투입한다.

### 전문가 선정

주제를 분석하여 적합한 전문가 3-5명을 추천. **사용자 확인 필수**:

```
지금까지 내용 기반으로 전문가 검토를 돌리겠습니다.

| 전문가 | 투입 이유 |
|--------|---------|
| @expert-strategy | 비즈니스 모델 검증 |
| @expert-pm-planner | MVP 스코프 적절성 |
| @expert-ui-ux-designer | UX 관점 핵심 플로우 |

진행할까요? (추가/제거 가능)
```

전문가 별칭은 `skills/expert-panel/references/panels.md` 참조.

### 디스패치

사용자 확인 후 각 전문가를 Agent 도구로 **병렬 호출**:

```
Agent 도구 호출:
  subagent_type: {에이전트 ID}
  prompt: |
    ## MVP 기획 방향성 검토 (1차)

    다음 MVP 아이디어에 대해 당신의 전문 분야 관점에서 검토해주세요.

    ### 서비스 개요
    - 한 문장 설명: {서비스 설명}
    - 타겟 사용자: {타겟}
    - 핵심 기능: {기능 목록}
    - 제약 조건: {제약}

    ### 요청 사항
    1. 이 방향에 대한 의견 (찬성/우려)
    2. 놓치고 있는 기능이나 관점이 있다면 제안
    3. MVP 스코프 조정 의견 (넣어야 할 것 / 빼야 할 것)
    4. 최신 트렌드 기반 인사이트 (WebSearch로 리서치 후)

    간결하게 답변해주세요 (500 토큰 이내).
```

배치 전략: 1-5명 동시 병렬, 6+명 5명씩 배치.

### 결과 요약

전문가별 핵심 제안을 사용자에게 제시:

```
## 1차 전문가 검토 결과

💡 전략 전문가: "구독 모델보다 건당 결제가 MVP에 적합"
💡 PM: "채팅은 MVP에서 빼고 매칭+예약에 집중 추천"
💡 UX: "온보딩 2단계면 충분, 소셜 로그인 필수"

→ 이 의견들 반영할까요? 취사선택 해주세요.
```

## Phase 3: 기획 심화

1차 전문가 피드백 중 사용자가 채택한 내용을 반영하여 대화를 이어간다.

- 전문가가 제안한 새 아이디어에 대해 사용자와 논의
- MVP 스코프 최종 확정
- 유저 플로우 확정
- KPI 논의 (Phase 1에서 다루지 않았으면 여기서)
- 충분히 구체화되면 Phase 4로 넘어감

---

## Phase 4: 기획 초안 작성

기획서와 플로우차트를 생성하여 사용자에게 제시한다.

### 기획서 형식

```markdown
# {서비스명} MVP 기획서

## 1. 서비스 개요
- 한 문장 설명
- 타겟 사용자
- 핵심 가치 (왜 이걸 써야 하는지)

## 2. MVP 스코프
### 포함 (Must-have)
- [ ] 기능 A — 설명
- [ ] 기능 B — 설명

### 제외 (Nice-to-have, 이후 버전)
- 기능 C — 이유
- 기능 D — 이유

## 3. 유저 플로우
### 메인 플로우
(Mermaid flowchart TD)

### 서브 플로우
- 플로우별 Mermaid 차트 또는 텍스트 설명

## 4. 전문가 인사이트
| 전문가 | 핵심 제안 | 반영 여부 |
|--------|---------|----------|
| 전략 | "..." | ✅ 반영 / ❌ 미반영 (사유) |

## 5. 제약 조건
- 기술 스택: ...
- 기간/예산: ...
- 플랫폼: ...

## 6. 성공 지표 (KPI)
- 지표 A: 목표값
- 지표 B: 목표값
```

### 플로우차트 가이드

Mermaid `flowchart TD` 문법으로 기획서에 포함:
- **메인 플로우**: 반드시 포함. 서비스의 핵심 사용자 여정.
- **서브 플로우**: Must-have 기능별로 별도 플로우차트 또는 텍스트 설명. 복잡도에 따라 조절.

---

## Phase 5: 2차 전문가 보강 (완성도 검증)

기획 초안 완성 후 전문가에게 최종 검토를 돌린다.

### 1차와 2차의 차이

| | 1차 (Phase 2) | 2차 (Phase 5) |
|---|---|---|
| 시점 | 윤곽만 잡힌 상태 | 기획 초안 완성 후 |
| 목적 | 방향성 검증 + 아이디어 | 완성도 검증 + 보강 |
| 깊이 | 가볍게 (의견 수준) | 깊게 (기획서 전체 리뷰) |
| 전달 내용 | 요약 정보 | 완성된 기획서 + 플로우차트 전문 |

### 전문가 선정

기획서 내용 변화에 따라 전문가 교체를 추천할 수 있음. 예: Phase 3에서 결제 기능이 추가되었으면 `@expert-security`를 추가 추천. **사용자 확인 필수**.

### 디스패치

```
Agent 도구 호출:
  subagent_type: {에이전트 ID}
  prompt: |
    ## MVP 기획서 최종 검토 (2차)

    다음 MVP 기획서를 당신의 전문 분야 관점에서 검토해주세요.

    ### 기획서 전문
    {기획서 마크다운 전체}

    ### 요청 사항
    1. 기획서에서 빠진 시나리오나 엣지 케이스
    2. 추가하면 좋을 아이디어
    3. 우선순위 조정 제안 (Must-have ↔ Nice-to-have 이동)
    4. 최신 트렌드 기반 개선점 (WebSearch로 리서치 후)

    간결하게 답변해주세요 (500 토큰 이내).
```

---

## Phase 6: 최종 산출물

사용자가 2차 전문가 피드백을 취사선택한 뒤 기획서를 확정한다.

### 저장 위치

```
docs/mvp/{서비스명-kebab-case}-기획서.md
```

서비스명 정규화: 한글 허용, 공백은 `-`로 치환, 특수문자 제거.
예: "반려동물 산책 매칭 앱" → `docs/mvp/반려동물-산책-매칭-앱-기획서.md`

확정 후 git 커밋.

---

## Phase 7: 다음 단계 선택

```
기획서가 확정되었습니다. 다음으로 뭘 할까요?

1. 📱 화면 설계 — 와이어프레임 수준 화면 구성
2. 🏗️ 기술 설계 — 아키텍처, DB 스키마, API 설계
3. 🚀 구현 — 스캐폴딩 + TDD 구현 + 코드 리뷰
4. ✅ 여기서 끝 — 기획서만 가져갈게요
```

| 선택 | 동작 | 산출물 |
|------|------|--------|
| 화면 설계 | 기획서의 유저 플로우 기반, Mermaid/ASCII로 화면 구성 제시. 완료 후 다시 메뉴. | `docs/mvp/{서비스명}-화면설계.md` |
| 기술 설계 | 기획서 기반 아키텍처+API+DB 설계서 생성. 완료 후 다시 메뉴. | `docs/mvp/{서비스명}-기술설계.md` |
| 구현 | `superpowers:writing-plans` 스킬로 전환하여 구현 플랜 생성 → 실행. | 구현 코드 |
| 끝 | 종료 | - |

복수 선택 가능 — 순차적으로 진행.

---

## 중단 및 재개

- 사용자가 중간에 중단하면 현재까지 내용을 `docs/mvp/{서비스명}-초안.md`에 저장
- 초안 파일이 있으면 "이전에 작업하던 기획이 있습니다. 이어서 할까요?" 제안
- 전문가 에이전트가 응답하지 않으면 해당 전문가 없이 진행, 결과에 "⏱ 응답 없음" 표기
